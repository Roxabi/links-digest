ascii-video is a skill i built for (and with) Hermes Agent that teaches coding agents how to produce experimental, generative ASCII video from scratch.

across 9 reference documents, roughly 7,000 lines of technical spec, it gives an agent everything it needs to write a single self-contained Python script that renders colored ASCII character video. audio/video/text/nothing input, MP4 or GIF output. the agent builds the entire renderer each time, adapted to the project.

it is built in to Hermes Agent by @NousResearch but works great with any coding agent that can read docs and write Python.

what it is

at a high level, the skill gives an agent the knowledge to do the following: take any monospace font on the system, pre-rasterize every character in a palette to a bitmap, lay out a grid of cells at that font size across the output resolution, compute a brightness value and a color for every cell using mathematical functions (sine fields, noise, simulations, signed distance fields, particle systems, or sampled video frames), pick a character from the palette based on brightness, tint the bitmap with the cell's color, and composite it onto a pixel canvas. do that at multiple grid densities (a fine grid for atmospheric texture, a standard grid for the main visual, a coarse grid for highlights) and blend the resulting canvases together with Photoshop-style blend modes. run a chain of post-processing shaders over the composited frame: bloom, CRT distortion, chromatic aberration, film grain, glitch effects, color grading. optionally blend with a transformed copy of the previous frame for temporal feedback (zoom trails, rotation echoes, hue-shifting ghosts). pipe the final frame to ffmpeg. repeat 24 times per second.

the skill documents 21 value field generators for computing per-cell brightness (trigonometric patterns, fractal noise, voronoi cells, reaction-diffusion, cellular automata, strange attractors, SDFs), 9 hue field generators for color, 24 character palettes spanning density ramps to braille to katakana to runes, 38 composable shaders, 20 pixel blend modes, 7 feedback buffer transforms, 16 mask types for spatial compositing, 9 particle systems, 11 coordinate transforms, temporal coherence tools (easing, keyframes, field morphing), 6 input modes (video, audio, generative, hybrid, lyrics, TTS narration), hardware auto-detection with 5 quality profiles, and a set of compositional design patterns for making scenes that look intentional rather than random.

the output is full RGB pixel frames at 1080p or 4K, encoded to H.264 or GIF via ffmpeg. the agent writes the entire renderer as a single Python file each time, selecting from the toolkit based on what the project needs.

the pipeline

every mode follows the same six stages: input, analyze, scene function, tonemap, shade, encode.

input loads source material: audio files decoded to mono WAV at 22050 Hz via ffmpeg, video frames sampled at the target FPS, images, SRT subtitle files, or nothing at all for generative mode.

analyze extracts per-frame features. for audio, this means a windowed FFT with hop size equal to sample rate divided by FPS and a 2x overlap Hanning window. the spectrum gets split into 6 frequency bands: sub (20-80 Hz), bass (80-250 Hz), lomid (250-500 Hz), mid (500-2000 Hz), himid (2-6 kHz), and hi (6 kHz+). from these i derive 16 features per frame: RMS energy, per-band power (6), spectral ratios (6, volume-independent), spectral centroid, spectral flatness, spectral flux, and beat detection. beat detection works by smoothing the flux signal with a convolution kernel, running scipy's find_peaks with height=0.15, distance=fps//5, and prominence=0.05, then generating an exponential decay signal (bdecay) that drops from 1.0 to 0 over roughly half a second after each beat. all features get EMA-smoothed (alpha=0.12 for slow features like centroid, alpha=0.3 for fast features like RMS) then normalized to [0, 1]. for video, the analyzer extracts per-frame brightness, contrast, edge density, motion magnitude, dominant hue, and color variance.

scene functions are where the visuals happen. each one receives four arguments: a renderer with lazy-initialized multi-density grid access, the feature dict, the current time (local to the scene, starting at 0.0), and a persistent state dict for simulations that evolve across frames. the function returns a raw pixel canvas - a uint8 numpy array at full output resolution. internally, a scene might render a background atmosphere on a dense xs grid (8px font, 400x108 cells at 1080p), a primary visual on a standard md grid (16px, 192x56), and sparse accent highlights on a coarse lg grid (20px, 160x45), compositing them with pixel blend modes. the scene function has full control over how many grids to use, what effects to run on each, and how to combine them.

tonemap handles the brightness problem that makes or breaks ASCII video. characters are small bright shapes on a black field; the vast majority of pixels in any frame are background. naive linear brightness multipliers either clip highlights to white or leave the midtones too dark. the tonemap function instead computes the 1st and 99.5th percentile of pixel values, normalizes the range, then applies a gamma curve. gamma below 1.0 lifts the midtones without clipping. default is 0.75, destructive effects like solarize get 0.55, posterize gets 0.50, already-bright scenes get 0.85. this runs after the scene function and before shaders.

shade applies two things in sequence: first the feedback buffer (if configured), which blends the current frame with a spatially transformed version of the previous frame for temporal recursion effects, then the shader chain, which runs an ordered sequence of post-processing shaders on the pixel canvas. both are configured per-scene in the scene table.

encode pipes raw RGB bytes to ffmpeg via stdin for H.264 encoding. each scene renders as a separate clip file in parallel across workers, then they get concatenated and audio gets muxed in.

modes

video-to-ASCII takes a video file and recreates it as colored ASCII. the source frames get downsampled, each pixel mapped to a character based on luminance and colored by sampling the original pixel's hue.

audio-reactive takes an audio file and generates visuals driven entirely by the extracted features. bass drives ring pulsation, beats trigger particle bursts, spectral centroid shifts hue, flux controls glitch intensity. each section of the video can use completely different effects and palettes.

generative mode takes no input. visuals come from pure math -- sine fields, noise, reaction-diffusion, attractors. a synthetic feature generator produces fake audio features (120 BPM beat signal, sinusoidal band energies) so the same scene functions work whether driven by real audio or not.

hybrid combines video source with audio-reactive overlays. the video provides a luminance/edge texture layer while audio features drive effects layered on top.

lyrics/text mode takes audio plus a timed SRT file and renders text with per-word effects: typewriter reveal, fade-in, flash, scatter, wave. text renders on coarse grids (lg or xl) while backgrounds and effects run on finer grids beneath.

TTS narration mode takes text quotes plus an ElevenLabs API key. it generates speech for each quote using a pool of 21 voices (George, Brian, Bill, Roger, Eric, Daniel, Charlie, Chris, Will, Liam, River, Sarah, Alice, Lily, Matilda, Laura, Harry, Bella, Callum, Jessica, Adam) assigned by deterministic shuffle. pronunciation can be controlled separately from display text for brand names and abbreviations. the audio pipeline generates per-quote WAV clips, concatenates them with computed silence gaps, then mixes with background music -- TTS centered mono-to-stereo, BGM loudness-normalized at 15% volume with extrastereo=m=2.5 widening, combined via amix. sync verification compares audio beat timestamps against visual beat timestamps with a 50ms tolerance, targeting mean drift under 20ms and max under 42ms (one frame at 24fps). each quote gets one of 8 rotating visual presets with a unique hue, accent color, background effect, and text color.

the grid system

characters render on fixed-size grids determined by font size. each grid pre-computes coordinate arrays at init: row indices (rr), column indices (cc), distance from center (dist), angle from center (angle), and normalized versions of each (dist_n, dx_n, dy_n). these get reused by every effect function without recomputation.

six density levels: xs at 8px (400x108 landscape, 225x192 portrait), sm at 10px (320x83, 180x148), md at 16px (192x56, 112x100), lg at 20px (160x45, 90x80), xl at 24px (137x37, 75x66), xxl at 40px (80x22, 45x39). the system supports landscape 1920x1080, portrait 1080x1920, square 1080x1080, ultrawide 2560x1080, and 4K variants.

every character in every palette gets pre-rasterized to a float32 bitmap at init using Pillow's truetype renderer. at render time, a scene function calls _render_vf() which maps a value field to characters via a palette lookup, computes per-cell colors from a hue field, then composites each character's bitmap onto the pixel canvas using the cell's color. the per-cell compositing loop is pure Python and runs at 80-150ms per frame -- this is the bottleneck. everything before and after it is vectorized numpy.

layering works because grids at different densities produce canvases at the same pixel resolution. a scene renders its background on a sm grid, its content on a md grid, and its accent on a lg grid, each producing a 1920x1080x3 uint8 array. these get composited with pixel blend modes (screen, add, colordodge, etc.) to produce the final scene canvas.

character palettes

24 named palettes ship with the skill, grouped by visual texture. density ramps give the classic ASCII art look. block elements (░▒▓█▄▀▐▌) produce chunky digital texture. braille patterns provide 32 discrete brightness levels in a pointillist style. stars (·✧✦✩✨★✶✳) sparkle. math symbols (·∘∙•°±×÷≈≠≡∞∫∑Ω) feel scientific. box drawing characters create circuit-board structure. katakana halfwidth characters do the matrix rain thing. Greek, runes, arrows, zodiac, alchemical, music notation -- each has a distinct feel.

there are also project-specific palettes: Hermes (mythology meets tech), ocean (wave characters), organic (botanical), machine (industrial). and per-project custom palettes are expected. every palette gets validated against the actual system font at init -- Pillow renders each character to a test image, and any glyph that produces a blank bitmap gets silently removed. this prevents invisible characters from eating grid cells.

font detection tries Menlo, Monaco, Courier, SF Mono, JetBrains Mono, Fira Code, IBM Plex, DejaVu Sans Mono in order. cell dimensions come from font.getmetrics() for ascent+descent (not textbbox, which returns wrong values on macOS Pillow).

value fields

value fields are functions that take a grid, features, time, and state, and return a float32 array in [0, 1] mapping every cell to a brightness value. they're the core creative primitive.

the 12 trigonometric fields cover the basics: vf_sinefield layers four sine products with configurable frequencies and speeds. vf_smooth_noise does multi-octave sine-based Perlin approximation. vf_rings generates concentric rings whose count responds to bass energy and whose edges wobble with mid frequencies. vf_spiral computes logarithmic spiral arms with configurable arm count and tightness. vf_tunnel creates infinite-depth perspective by inverting distance from center. vf_vortex twists angle by distance. vf_interference overlaps N directional sine waves creating moire patterns, each driven by a different audio band. vf_aurora makes horizontal flowing bands. vf_ripple generates concentric waves from configurable source points. vf_plasma sums sines at multiple orientations and speeds. vf_diamond makes checkerboard patterns. vf_noise_static is random per-cell per-frame.

the noise fields bring organic quality. vf_noise implements value noise without axis-aligned artifacts. vf_fbm stacks octaves of noise for clouds, terrain, and smoke - each octave doubles frequency and halves amplitude. vf_domain_warp uses the Inigo Quilez technique: evaluate fBM to get warp offsets, then evaluate fBM again at the warped coordinates. this produces flowing, impossible-looking organic forms. vf_voronoi scatters seed points that drift over time, computes nearest and second-nearest distances per cell, and offers three output modes: distance (cell interiors), edge (cell boundaries), and cell_id (per-cell index for coloring).

the simulation fields run stateful processes that evolve across frames. vf_reaction_diffusion implements Gray-Scott with Laplacian diffusion - two chemical species, U and V, where U feeds V and V kills itself, computed via numpy roll operations for the discrete Laplacian. 7 preset parameter pairs (feed rate, kill rate) produce coral branching, spots, worms, labyrinths, mitosis, pulsating blobs, and chaos. vf_game_of_life runs Conway's rules with analog trail decay at 0.92 per frame, plus 4 alternate rule sets. vf_strange_attractor iterates Clifford (sin/cos), De Jong (sin), or Bedhead (sin with conditional) equations across thousands of points, bins them into a histogram, and log-normalizes. vf_temporal_noise uses 3D noise that morphs smoothly in the time dimension without drifting in any spatial direction.

the SDF system provides geometric primitives via signed distance functions. sdf_circle, sdf_box, sdf_ring, sdf_line, sdf_triangle, sdf_star, sdf_heart each return a distance field. these combine with smooth boolean operations: sdf_smooth_union blends two shapes together with a rounding parameter k, sdf_smooth_subtract carves one from another with a rounded edge. sdf_repeat tiles any primitive infinitely. two renderers convert distance to value: sdf_render for solid shapes with configurable edge softness, sdf_glow for outlines that glow brightest at the boundary.

hue fields and color

hue fields are factory functions that return closures with the same (g, f, t, S) signature as value fields but producing hue values in [0, 1]. hf_fixed returns a constant. hf_angle maps position angle from center to a rainbow. hf_distance maps radial distance to hue for depth effects. hf_time_cycle rotates hue uniformly over time. hf_audio_cent follows the audio spectral centroid for timbral color shifting. hf_gradient_h and hf_gradient_v produce directional gradients. hf_plasma generates organic hue variation. hf_oklch_angle produces a perceptually uniform rainbow using the OKLCH color space.

the color system goes beyond basic HSV. OKLAB/OKLCH support includes a full sRGB-to-linear-to-OKLAB conversion pipeline, perceptually uniform gradient interpolation (lerp_oklab, lerp_oklch), and algorithmic color harmony generation -- complementary (opposite hue), triadic (three equidistant hues), analogous (neighboring hues), split-complementary (opposite plus two neighbors), and tetradic (four equidistant). 10 discrete RGB palettes cover common aesthetics: neon, pastel, mono green, mono amber, cyberpunk, vaporwave, earth, ice, blood, forest.

shaders

38 shaders operate on the pixel canvas, applied in sequence by a ShaderChain. each shader is a function that takes the canvas, features, time, and optional keyword arguments.

geometry shaders: crt does barrel distortion with a cached remap table. pixelate reduces effective resolution to configurable block size. wave_distort applies sinusoidal row or column displacement. kaleidoscope remaps to polar coordinates and folds by N. four mirror modes -- horizontal, vertical, quad, and diagonal -- each reflect different axes.

channel shaders: chromatic shifts R and B channels horizontally by an amount that scales with bdecay (beat-reactive). channel_shift moves each channel independently on x and y. channel_swap reorders RGB. rgb_split_radial pushes channels outward from center.

color shaders: invert negates. posterize quantizes to N levels. threshold goes binary. solarize inverts pixels above a threshold (the darkroom effect). hue_rotate shifts all hues. saturation scales. color_grade applies per-channel RGB multipliers. color_wobble modulates each channel with time-varying sines scaled by RMS. color_ramp maps luminance through a custom gradient (useful for fire: black -> red -> orange -> yellow -> white).

glow and blur: bloom extracts pixels above a threshold, 4x downsamples, box-blurs, screen-blends back -- runs in 84ms vs 424ms for a scipy gaussian approach. edge_glow does edge detection and overlays colored edges. soft_focus blends with a blurred copy. radial_blur zooms from center.

noise: grain generates 2x-downsampled random noise tiled up (1/4 the generation cost), scaled by RMS. static is full-resolution random noise.

patterns: scanlines darken every Nth row. halftone overlays a dot pattern.

tone: vignette darkens edges using a cached distance field, clamped to a minimum of 0.15 so it never fully blacks out. contrast adjusts around midpoint 128. gamma corrects. levels does Photoshop-style black/white/midtone. brightness is a straight multiplier.

glitch: glitch_bands displaces horizontal rows by random amounts, count and magnitude scaling with bdecay and sub bass. block_glitch copies and displaces random rectangular regions. pixel_sort sorts contiguous bright pixel runs by luminance. data_bend displaces raw bytes (datamosh artifacts).

9 mood presets chain these together: retro terminal is CRT + scanlines + grain + amber tint. cinematic is bloom + vignette + grain + color grade. glitch art is heavy chromatic + glitch bands + color wobble + pixel sort. psychedelic is color wobble + chromatic + kaleidoscope + feedback with hue shift. data corruption is pixel sort + data bend + block glitch + posterize. there are also clean modern, dreamy, harsh industrial, and recursive/infinite presets.

11 tint presets apply per-channel multipliers: warm (1.15, 1.0, 0.85), cool (0.85, 0.95, 1.15), matrix green (0.7, 1.2, 0.7), amber (1.2, 0.9, 0.6), sepia (1.2, 1.05, 0.8), neon pink (1.3, 0.7, 1.1), ice (0.8, 1.0, 1.3), blood (1.4, 0.7, 0.7), forest (0.8, 1.15, 0.75), void (0.85, 0.85, 1.1), sunset (1.3, 0.85, 0.7).

blend modes and composition

20 pixel blend modes implemented as numpy lambda functions operating on float32 [0, 1] arrays. screen is 1-(1-a)*(1-b) for additive brightening. multiply darkens. overlay branches on a < 0.5. colordodge is a/(1-b+epsilon) for extreme brightening at overlap. difference is abs(a-b). exclusion is a+b-2ab. all 20 are available in sRGB space; a separate blend_canvas_linear() converts to linear light, blends, and converts back.

the feedback buffer stores the previous frame's output and applies a spatial transform before blending it into the current frame. 7 transforms: zoom crops inward and upscales (creates expanding tunnel effect), shrink pads edges and downscales center, rotate_cw and rotate_ccw apply small per-frame rotation via affine transform, shift_up and shift_down scroll vertically with black fill, mirror_h flips horizontally. an optional hue_shift parameter rotates the hue of the entire feedback buffer each frame, creating rainbow trail effects. the blend itself uses any of the 20 blend modes at configurable opacity and decay rate.

the masking system provides 16 mask types for spatial compositing. shape masks: circle, rectangle, ring, horizontal gradient, vertical gradient, radial gradient, each with optional feathered edges. procedural masks: any value field output thresholded to become a mask, and text stencils rendered via Pillow. animated masks: iris (circle expanding from 0 to max radius), horizontal wipe, vertical wipe, noise dissolve (threshold sweeps through a noise field). boolean operations: union, intersection, subtraction, inversion. masks apply to canvases (pixel-level alpha) or to value fields (spatial blending of two effects).

particles

9 particle system types, all persistent across frames via the state dict. beat-triggered explosions emit radial bursts from center on beat onset with per-particle gravity and life decay. rising embers drift upward from the bottom edge with horizontal jitter and fade. dissolving clouds spread outward with accelerating fade. 3D projected starfields have z-depth stars approaching the camera with streak trails. orbital particles follow circular/elliptical paths around center points. gravity wells attract particles toward configurable attractor positions. boid flocking implements separation, alignment, and cohesion with a spatial hash for O(n) neighbor queries instead of O(n^2). flow-field followers are steered by the gradient of any value field -- the field acts as a vector field pushing particles along. trail particles draw fading lines between their current and previous positions.

14 themed particle character sets: energy, spark, leaf, snow, rain, bubble, data, hex, binary, rune, zodiac, dot, dash, and more.

coordinate transforms

11 UV-space transforms warp the coordinate grid before effect evaluation. uv_rotate rotates around center. uv_scale zooms in or out. uv_skew applies horizontal or vertical shear. uv_tile repeats the coordinate space N times with optional mirror seaming at edges. uv_polar converts cartesian to polar (makes any linear effect radial). uv_cartesian_from_polar does the inverse. uv_twist applies rotation that increases with distance from center, creating spiral distortion. uv_fisheye applies barrel/pincushion distortion. uv_wave displaces coordinates sinusoidally. uv_mobius applies a Mobius conformal transformation -- complex plane inversion that creates impossible-looking inside-out geometry. make_tgrid wraps any pair of transformed coordinate arrays into a grid object that effect functions can use directly.

temporal coherence

10 easing functions: linear, in/out/in-out quad, in/out/in-out cubic, in/out expo, elastic, bounce. keyframe() defines parameter values at specific times with eased interpolation between them. keyframe_array() does the same but with numpy arrays as values (for interpolating entire fields). vf_morph() crossfades between two value fields over a configurable time range. vf_sequence() cycles through a list of value fields with crossfade duration between each. vf_temporal_noise() generates 3D noise that evolves smoothly in the time axis without drifting spatially -- the visual pattern changes in place rather than scrolling.

design patterns

the part i care about most. the toolkit above is large. the design patterns are what keep it from turning into a random effects generator.

layer hierarchy: every scene should have a background layer (dim, dense grid, atmosphere), a content layer (the main visual, standard grid, full brightness), and an accent layer (sparse, coarse grid, highlights that punctuate without competing). the background is mood. the content is what the scene is about. the accent is emphasis.

directional parameter arcs: the defining parameter of each scene should go somewhere over its duration. a ring that expands from center to edge over 5 seconds has a direction. a vortex twist that ramps from gentle to extreme has a story. an arm count that increases from 2 to 8 builds complexity. sin(t * N) oscillations are fine for secondary parameters like saturation shimmer, but the main parameter should use progress-based formulas: linear ramp, ease-out (fast start, gentle finish), ease-in (slow start, accelerating), step reveal (nothing until a threshold, then fade in), or build-and-plateau.

scene concepts: each scene should be a visual idea, not an effect name. "emergence" - a ring of light expands and ASCII characters populate behind it. "descent" - a tunnel plunge with accelerating speed. "collision" - two wave fronts converge and interfere at the meeting point. "entropy" - clean geometric patterns consumed by reaction-diffusion. the name tells you what happens. the layers, palettes, feedback transforms, and parameter arcs all serve that concept.

additional patterns i documented: counter-rotating dual systems where two spirals or vortexes rotate in opposite directions creating constantly shifting interference. wave collision where two wavefronts approach from opposite sides and produce maximum interference at the meeting point. progressive fragmentation where voronoi cell count ramps from 8 to 38 over 5 seconds, edges glow brighter as they multiply. entropy where a clean geometric pattern (rings, SDF shapes) fades out while reaction-diffusion grows in over the same duration. staggered layer entry (crescendo) where layers enter one at a time -- first plasma alone, then spiral joins after 2 seconds, then vortex, then interference, then rings, then fBM, then a sinefield wash - building to overwhelming density over 15 seconds.

scene ordering for multi-scene reels: vary mood between adjacent scenes (don't put two calm ones next to each other), randomize order rather than grouping by type, open with energy, end on the strongest scene.

these patterns came from building a highlight reel twice. the first version had 20 scenes with random effect combos, oscillating parameters, and scenes named after their effects. it looked like a tech demo. i rewrote every scene with a concept, a directional arc, a layer hierarchy, and motivated choices for palettes and feedback transforms. same toolkit. the second version looked more like art.

hardware adaptation

the skill auto-detects hardware via multiprocessing.cpu_count() for cores, sysctl on macOS or /proc/meminfo on Linux for RAM, and shutil.which for ffmpeg. worker count gets capped at cores minus 1 (leaving headroom for the OS) and further capped at 60% of RAM divided by 150MB per worker.

five quality profiles: draft (960x540 at 12fps, minimal shaders, 200 max particles), preview (1280x720 at 15fps, standard shaders, 500 particles), production (1920x1080 at 24fps, full shaders, 1200 particles), max (3840x2160 at 30fps, full everything, 3000 particles), and auto (estimates total render time at 180ms per frame per worker, downgrades to 720p if it would exceed one hour, drops to 720p/15fps on machines with under 4GB RAM or 2 or fewer cores).

the per-frame budget: feature extraction takes 1-5ms (pre-computed for all frames up front), effect functions take 2-15ms (all vectorized numpy), character rendering takes 80-150ms (the Python per-cell loop -- the only non-vectorized part), shader pipeline takes 5-25ms depending on which shaders are active, and ffmpeg encoding is roughly 5ms amortized by pipe buffering. total is 100-200ms per frame. at production quality with 8 workers, 30 seconds of video (720 frames) renders in about 2 minutes. 5 minutes (7200 frames) takes about 17 minutes. 4K roughly quadruples these.

extending

the reference docs are the interface. adding a new value field means documenting its signature, parameters, and behavior in effects.md. adding a new shader means adding its implementation pattern to shaders.md. the agent reads whatever's in the references and builds from it. custom palettes, color strategies, particle types, and shader chains are expected per-project. the skill documents the vocabulary and the composition patterns. the agent combines them differently every time.
